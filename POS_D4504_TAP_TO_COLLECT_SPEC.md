# คำสั่งพัฒนา: หน้าจอ POS iMin D4-504 — แตะบัตรรับอาหาร Pre-Order

โปรเจกต์: Wise Canteen / UPOS
อุปกรณ์: iMin D4-504 (จอคู่ — จอหลักฝั่งนักเรียน + customer display)
ธีม: ใช้ design system เดิมของ Wise Canteen (primary #1264E3, two-weight typography)

---

## 1. ภาพรวม flow

```
[หน้ารอแตะบัตร] → แตะบัตร → [ตรวจสอบสถานะ] → แสดง 1 ใน 13 เคส →
  - ถ้าสถานะปกติ → [รับอาหารสำเร็จ] หรือ [ยืนยันจากหน้ารายละเอียด]
  - ถ้าแตะซ้ำ → [แจ้งเตือนรับไปแล้ว]
  - ถ้าเป็น edge case → แสดงข้อความเฉพาะเคส (ดูข้อ 4)

[หน้ารอแตะบัตร] → กดปุ่ม "ประวัติการจอง" → [หน้าประวัติการจองรายบุคคล]
  → filter วันที่ / รอบเวลา (เช้า-กลางวัน-เย็น-ทั้งหมด)
  → กดแถว → [modal รายละเอียด] → ปุ่มยกเลิกการจอง (ต้องกรอกเหตุผล + รหัส supervisor)

[หน้าภาพรวม staff] (แยก entry point จากฝั่งนักเรียน)
  → filter วันที่ / รอบเวลา / ค้นหาชื่อ-ชั้น
  → การ์ดสรุป: จองทั้งหมด / มารับแล้ว / ไม่มารับ
  → list นักเรียนทั้งหมด → กดแถว → modal รายละเอียดเดียวกับด้านบน
```

## 2. สถานะการจอง (5 สถานะ)

| Status (EN) | Label ที่แสดง | สี | ความหมาย |
|---|---|---|---|
| Confirmed | จองแล้ว | success (เขียว) | นักเรียนจองไว้ล่วงหน้า ยังไม่ถึงวัน |
| Ready to collect | รอวันรับ | accent (น้ำเงิน) | ถึงคิวแล้วแต่ยังไม่ถึงวันรับจริง |
| Collected | รับแล้ว | muted (เทา) | นักเรียนมาแตะบัตรรับอาหารแล้ว |
| Missed | ไม่มารับ | danger (แดง) | ถึงวันแล้วแต่ไม่มาแตะบัตรรับ (auto เมื่อพ้น cutoff) |
| Cancelled | ยกเลิกแล้ว | danger (แดง) | ถูกยกเลิก ต้องมี reason + ผู้ยืนยัน |

> หมายเหตุ: mapping นี้เป็นการเรียงลำดับที่ทีมยืนยันไว้ในแชท ถ้ามีการเปลี่ยนแปลงให้แก้ตารางนี้เป็นจุดอ้างอิงเดียว (single source of truth)

## 3. หน้าจอหลัก (จอ 1 — student-facing)

### 3.1 Idle
- ข้อความ "แตะบัตรเพื่อรับอาหาร" + ไอคอนบัตร
- แสดงมื้อ/รอบเวลาปัจจุบัน
- ปุ่ม "ประวัติการจอง" มุมล่างขวา → ไปหน้า 3.5

### 3.2 แสดงรายการ + สถานะ (หลังแตะบัตรปกติ)
- รูป/อวตาร + ชื่อ + ชั้นเรียน
- badge สถานะ (ใช้สีตามข้อ 2)
- รายการอาหาร Pre-Order (ชื่อเมนู, จำนวน, หมายเหตุแพ้อาหาร)
- ปุ่ม "ยืนยันรับอาหาร" (แสดงเมื่อสถานะเป็น Confirmed หรือ Ready to collect เท่านั้น)

### 3.3 รับอาหารสำเร็จ
- ไอคอนติ๊กถูกสีเขียว
- ข้อความ: "รับอาหารสำเร็จ"
- ชื่อนักเรียน + เมนูที่รับ + เวลา
- auto กลับหน้า idle ใน 3 วินาที

### 3.4 แตะซ้ำ (สถานะ Collected อยู่แล้ว)
- ไอคอนเตือนสีเหลือง (ไม่ใช่แดง — ไม่ใช่ข้อผิดพลาดร้ายแรง)
- ข้อความ: "รับอาหารไปแล้ว"
- รายละเอียด: "{ชื่อนักเรียน} รับไปแล้วเมื่อ {เวลา}"
- ท้าย: "หากมีข้อสงสัย ติดต่อพนักงานโรงอาหาร"
- auto กลับหน้า idle ใน 3 วินาที

### 3.5 หน้าประวัติการจอง (รายบุคคล)
- ช่องเลือกช่วงวันที่ (from–to)
- tab filter รอบเวลา: ทั้งหมด / รอบเช้า / รอบกลางวัน / รอบเย็น (ดึงชื่อ tab จาก config รอบเวลาที่ตั้งไว้ในระบบ)
- list รายการ grouped by วันที่ พร้อม badge สถานะ
- กดแถว → เปิด modal รายละเอียด (ข้อ 5)

## 4. Edge cases ตอนแตะบัตร (แสดงแทนหน้า 3.2)

| # | เคส | ข้อความหลัก | ข้อความรอง | ระดับความสำคัญ (สี) |
|---|---|---|---|---|
| 1 | ไม่มีการจองเลยวันนี้ | ไม่พบรายการจอง | "{ชื่อ} ยังไม่ได้จองมื้อนี้" | เทา (info) |
| 2 | จองไว้คนละมื้อ | จองไว้มื้อ{ชื่อมื้อ} | "รับได้ {ช่วงเวลา}" | เหลือง (warning) |
| 3 | มาก่อนเวลาเปิดรับ | ยังไม่ถึงเวลารับ | "เปิดรับ {เวลาเริ่ม} น." | น้ำเงิน (info) |
| 4 | พ้นเวลารับแล้ว (auto → Missed) | พ้นเวลารับแล้ว | ปุ่ม "แจ้งพนักงานให้ override" | แดง (danger) |
| 5 | บัตรไม่ผูกกับผู้ใช้ในระบบ | ไม่พบข้อมูลบัตรนี้ | "กรุณาติดต่อพนักงาน" | เทา (info) |
| 6 | บัตรถูกระงับ/แจ้งหาย | บัตรถูกระงับใช้งาน | "กรุณาติดต่อพนักงาน" | แดง (danger) |
| 7 | ออเดอร์ถูกยกเลิกไปแล้ว | การจองนี้ถูกยกเลิกแล้ว | "กรุณาติดต่อพนักงาน" | แดง (danger) |
| 8 | ยังไม่ชำระเงิน (ถ้าระบบผูก payment) | ยังไม่ชำระเงิน | "แจ้งพนักงานก่อนรับอาหาร" | เหลือง (warning) |
| 9 | ระบบออฟไลน์/sync ช้า | กำลังตรวจสอบข้อมูล... | ถ้า timeout → "เชื่อมต่อไม่ได้ ลองใหม่อีกครั้ง" | เทา (loading/info) |

ทุกเคสต้อง auto กลับหน้า idle หลัง 4-5 วินาที ยกเว้นเคส 4 และ 8 ที่ต้องรอ action จากพนักงาน

## 4.5 เคสแตะบัตรไม่ผ่าน (ระดับฮาร์ดแวร์/การอ่านบัตร)

คนละกลุ่มกับข้อ 4 — เคสพวกนี้คือ "อ่านบัตรไม่สำเร็จ" ไม่ใช่ "อ่านสำเร็จแต่ข้อมูลมีปัญหา"

| # | เคส | จอ 1 (student-facing) | จอ 2 (customer display) |
|---|---|---|---|
| A | อ่านบัตรไม่สำเร็จ (แตะเอียง/เร็วไป) — ครั้งที่ 1-2 | "แตะบัตรอีกครั้ง" (ไอคอน refresh สีฟ้า) auto กลับหน้ารอใน 2-3 วิ | คงหน้า idle ไว้เฉยๆ ไม่ต้องแสดงอะไรเพิ่ม |
| B | อ่านไม่ผ่านซ้ำจนครบ retry limit (เช่น 3 ครั้ง) — น่าจะเป็นบัตรชำรุด | "อ่านบัตรไม่ได้ กรุณาติดต่อพนักงาน" (ไอคอนบัตรสีเหลือง) ไม่ auto-retry ต่อ | fallback กลาง "กรุณาติดต่อพนักงาน" |
| C | ตัวอ่านบัตร (NFC/RFID module) ขัดข้องทั้งเครื่อง | "เครื่องอ่านบัตรขัดข้อง กรุณาติดต่อพนักงาน" (ไอคอนประแจสีแดง) ควรส่ง alert ไปแจ้งพนักงาน/แอดมินอัตโนมัติด้วยถ้าทำได้ | fallback กลาง "กรุณาติดต่อพนักงาน" |
| D | แตะบัตรซ้ำระหว่างเครื่องกำลังประมวลผลรายการก่อนหน้า (debounce) | "กำลังดำเนินการ กรุณารอสักครู่" (ไอคอน loading) กันไม่ให้ยิง request ซ้ำ | คงหน้าเดิมค้างไว้จนประมวลผลเสร็จ |

หมายเหตุการพัฒนา:
- เคส A ควรนับ retry count ใน session เดียว ถ้าเกิน limit ให้สลับไปเคส B โดยอัตโนมัติ ไม่ต้องให้แตะไม่รู้จบ
- เคส C ควรแยก error code จาก backend/hardware layer ออกจากเคส B ให้ชัดเจน เพื่อให้พนักงานรู้ว่าต้องแก้ที่เครื่องไม่ใช่ที่บัตรนักเรียน
- เคส D คือ UI-level debounce ล้วนๆ ไม่ต้อง query backend ซ้ำ

## 5. Modal รายละเอียดการจอง (ใช้ร่วมกันทั้งฝั่งนักเรียนและ staff)

แสดง:
- ชื่อนักเรียน, ชั้น, วันที่, มื้อ/รอบเวลา, เวลารับจริง (ถ้ารับแล้ว)
- รายการอาหาร + หมายเหตุแพ้อาหาร
- badge สถานะปัจจุบัน

ส่วนยกเลิกการจอง (แสดงเมื่อสถานะยังไม่ถึง Collected):
- dropdown เหตุผล: นักเรียนลาป่วย / ผู้ปกครองยกเลิกล่วงหน้า / สั่งอาหารผิดรายการ / อื่นๆ (ระบุเพิ่มเติม)
- ช่องกรอกรหัส supervisor (type=password, 4-6 หลัก)
- ปุ่ม "ยืนยันยกเลิกการจอง" — disabled จนกว่าจะเลือกเหตุผลและกรอกรหัสครบ
- ถ้ารหัสผิด → แสดง error ใต้ช่องกรอก ไม่ปิด modal
- เมื่อยกเลิกสำเร็จ → status เปลี่ยนเป็น Cancelled พร้อม log เหตุผล + ผู้อนุมัติ + เวลา

## 6. หน้าภาพรวม (จอฝั่ง staff — คนละ entry point จากฝั่งนักเรียน)

- ช่องเลือกช่วงวันที่ (from–to)
- ช่องค้นหาชื่อนักเรียน/ชั้น (filter เสริม ไม่ใช่ primary view)
- tab filter รอบเวลา เหมือนข้อ 3.5
- การ์ดสรุป 3 ตัวเลข: จองทั้งหมด / มารับแล้ว / ไม่มารับ (คำนวณตามช่วงวันที่ + รอบที่เลือกอยู่)
- list นักเรียนทั้งหมดในช่วงที่ filter พร้อม badge สถานะ
- กดแถว → เปิด modal ข้อ 5

## 7. จอ 2 — Customer display (อ่านอย่างเดียว)

- sync กับจอหลักแบบ real-time
- Idle: โลโก้ + ข้อความต้อนรับ
- หลังแตะบัตร: ชื่อ + รายการอาหารแบบย่อ (ไม่มีปุ่มกด)
- รับสำเร็จ: ข้อความขอบคุณ + เมนูที่ได้รับ
- แสดง edge cases ให้ลูกค้ารับรู้ด้วย (ไม่ใช่ fallback กลางทั้งหมด) ตามตารางนี้:

| # | เคส | ข้อความจอ 2 |
|---|---|---|
| 1 | ไม่พบการจอง | "ไม่พบรายการจอง" |
| 2 | จองไว้คนละมื้อ | "จองไว้มื้อ{ชื่อมื้อ}" + "รับได้ {ช่วงเวลา}" |
| 3 | มาก่อนเวลา | "ยังไม่ถึงเวลารับ" + "เปิดรับ {เวลา} น." |
| 4 | พ้นเวลารับแล้ว | "พ้นเวลารับแล้ว" + "กรุณาติดต่อพนักงาน" |
| 5 | บัตรไม่รู้จัก | "ไม่พบข้อมูลบัตรนี้" + "กรุณาติดต่อพนักงาน" |
| 6 | บัตรถูกระงับ | "บัตรถูกระงับใช้งาน" + "กรุณาติดต่อพนักงาน" |
| 7 | ออเดอร์ถูกยกเลิก | "การจองนี้ถูกยกเลิกแล้ว" + "กรุณาติดต่อพนักงาน" (ไม่แสดงเหตุผลการยกเลิก) |
| 8 | ยังไม่ชำระเงิน | "ยังไม่ชำระเงิน" + "แจ้งพนักงานก่อนรับอาหาร" |
| 9 | ระบบออฟไลน์/sync ช้า | "กำลังตรวจสอบข้อมูล..." (ไม่ auto กลับ idle จนกว่าจะเชื่อมต่อได้) |

หลักการ: จอ 2 แสดงได้เกือบทุกเคสตรงๆ ยกเว้นรายละเอียดที่ละเอียดอ่อน (เช่น เหตุผลการยกเลิก, error code ทางเทคนิค) ให้ตัดเหลือแค่ "กรุณาติดต่อพนักงาน"

## 8. สิ่งที่ต้องยืนยันกับทีมก่อนพัฒนา

- [x] cutoff time ของแต่ละรอบ (เช้า/กลางวัน/เย็น) ตั้งค่าที่ไหน — **ตอบแล้ว:** ตั้งค่าผ่านระบบ backoffice (นอก repo นี้) — ค่าที่ใช้ในโค้ด mock ปัจจุบันเป็น placeholder รอเชื่อมค่าจริง
- [x] เคส 4 (พ้นเวลาแล้วขอ override) — ใครมีสิทธิ์กด และต้องขอรหัสเหมือนยกเลิกไหม — **ตอบแล้ว:** ไม่ต้องขอรหัส supervisor — กดแล้วเปลี่ยนสถานะเป็นรับแล้วทันที (**ต่างจาก assumption เดิมที่ Phase 1 เคยเขียนไว้ว่าใช้ PIN ร่วมกับยกเลิก — ต้องแก้ตอน Phase 3**)
- [x] เคส 8 (ยังไม่ชำระเงิน) เกี่ยวข้องกับระบบนี้จริงไหม หรือเช็ก payment แยกส่วน — **ตอบแล้ว:** เกี่ยวข้องจริง ผูกกับระบบเครดิต/กระเป๋าเงิน (wallet) ของนักเรียน — mock ปัจจุบันใช้ field `paid` แทนไปก่อนเพราะยังไม่มี backend เชื่อมจริง
- [ ] ระยะเวลา auto-dismiss ที่เหมาะสมของแต่ละหน้า (ปัจจุบันใช้ 3-5 วิ เป็นค่าตั้งต้น) — ยังไม่ถาม
