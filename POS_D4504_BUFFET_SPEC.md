# คำสั่งพัฒนา: หน้าจอ POS iMin D4-504 — เมนูบุฟเฟต์ (Walk-in, ไม่มีการจอง)

โปรเจกต์: Wise Canteen / UPOS
อุปกรณ์: iMin D4-504 (จอคู่)
ใช้ token เดียวกับ `POS_D4504_DESIGN_SYSTEM.md`

> **สำคัญ:** ไฟล์นี้เป็นคนละ flow จาก `POS_D4504_TAP_TO_COLLECT_SPEC.md` (Pre-Order) ห้ามนำสอง flow มาผสมกันในโค้ด บุฟเฟต์ไม่มีการจองล่วงหน้า ไม่มี 5 สถานะการจอง (Confirmed/Ready to collect/Collected/Missed) — เป็น transaction ทันทีแบบเดียวกับ top-up/purchase

## 1. ความแตกต่างจาก Pre-Order

| | Pre-Order | บุฟเฟต์ |
|---|---|---|
| ต้องจองล่วงหน้า | ใช่ | ไม่ต้อง — walk-in แตะบัตรจ่ายเลย |
| สถานะการจอง | มี 5 สถานะ | ไม่มี |
| ยอดเงิน | ตามเมนูที่สั่ง | ตามระดับชั้นของบัตรที่แตะ (ไม่โชว์ยอดก่อนแตะ) |
| หน้าประวัติ | ประวัติการจอง (badge สถานะ) | ประวัติการจ่ายบุฟเฟต์ (นับจำนวนคน+ยอดเงิน แยกตามชั้น) |

## 2. Flow

```
[หน้าแตะบัตร - บุฟเฟต์] → แตะบัตร → ระบบอ่านระดับชั้นจากบัตร → คำนวณยอดตามระดับชั้น
  → [ชำระเงินสำเร็จ] แสดงชื่อ+ระดับชั้น+ยอดที่จ่าย

[หน้าภาพรวมบุฟเฟต์] (staff)
  → filter วันที่
  → การ์ดสรุป: จำนวนคนทั้งหมด + ยอดรวม
  → แยกตามประเภท/ระดับชั้น (แต่ละแถวโชว์ ราคา/คน, จำนวนคน, ยอดรวม)
  → กดแถว → [drill-down] รายชื่อนักเรียนในระดับชั้นนั้น (ชื่อ + ระดับชั้น ไม่มี /ห้อง, ไม่มีเลขบัตร, ไม่มีอวตาร) + เวลาที่แตะบัตร
  → กดรายชื่อ (ถ้าต้องการยกเลิกรายการ) → [modal ยกเลิก]
```

## 3. หน้าจอ 1 (student-facing) — บุฟเฟต์

### 3.1 ก่อนแตะบัตร
- Badge ประเภทอาหาร "บุฟเฟต์" มุมซ้ายบน
- ไม่แสดงยอดเงิน (เพราะขึ้นกับระดับชั้น) — ข้อความ "ยอดขึ้นกับระดับชั้นของนักเรียน"
- ปุ่ม "ประวัติ" มุมขวาบน (สำหรับ staff เข้าดูภาพรวม ไม่ใช่ประวัติส่วนตัวนักเรียน)
- ข้อมูลซ้าย: เลขบัตร (ยังว่าง), ชื่อนักเรียน (ยังว่าง), วันที่

### 3.2 หลังแตะบัตรสำเร็จ
- ไอคอนติ๊กถูกสีเขียว
- "ชำระเงินสำเร็จ"
- ชื่อนักเรียน + ระดับชั้น + ประเภท "บุฟเฟต์"
- ยอดที่คำนวณได้ตามระดับชั้น (ตัวเลขใหญ่ สีธีมหลัก)

### 3.3 Edge cases ที่ใช้ร่วมกับ Pre-Order ได้ (ใช้ mapping เดียวกัน)
- บัตรไม่รู้จัก / บัตรถูกระงับ / ระบบออฟไลน์ / อ่านบัตรไม่ผ่าน (ฮาร์ดแวร์) — ใช้ข้อความและสีเดียวกับที่กำหนดใน `POS_D4504_TAP_TO_COLLECT_SPEC.md` ข้อ 4 และ 4.5 ได้เลย ไม่ต้องออกแบบใหม่

### 3.4 Edge case เฉพาะบุฟเฟต์
- **ยอดเงินในบัตรไม่พอจ่าย** → "ยอดเงินไม่เพียงพอ" + แสดงยอดที่ต้องจ่ายกับยอดคงเหลือ + "กรุณาเติมเงินหรือติดต่อพนักงาน"
- **แตะบัตรซ้ำในมื้อเดียวกัน** (กันจ่ายซ้ำ 2 ครั้งโดยไม่ได้ตั้งใจ) → เตือน "มื้อนี้ชำระบุฟเฟต์ไปแล้ว" พร้อมเวลา และถามยืนยันถ้าต้องการจ่ายซ้ำจริง (เช่น กรณีมีคนอื่นมากินด้วยบัตรเดียวกัน — ควรถามทีมว่าเคสนี้เกิดขึ้นได้จริงไหม)

## 4. จอ 2 (customer display) — บุฟเฟต์

- ก่อนแตะ: ข้อความต้อนรับ ไม่โชว์ยอด
- หลังแตะสำเร็จ: "ขอบคุณค่ะ" + ยอดที่จ่าย (ไม่ต้องโชว์ระดับชั้น/ชื่อเต็ม เพื่อความเป็นส่วนตัวระหว่างต่อคิว)
- edge case ใช้ fallback กลาง "กรุณาติดต่อพนักงาน" เหมือนกับ Pre-Order

---

## 5. หน้าภาพรวมบุฟเฟต์ (staff)

- ช่องเลือกช่วงวันที่
- การ์ดสรุป: จำนวนคนเข้าบุฟเฟต์ทั้งหมด + ยอดรวม
- แยกตามประเภท/ระดับชั้น: แต่ละแถวโชว์ ชื่อประเภท, ราคา/คน, จำนวนคน, ยอดรวมของประเภทนั้น
- กดแถว → drill-down แสดงรายชื่อนักเรียนที่เข้าประเภทนั้น:
  - แสดง: ชื่อนักเรียน + ระดับชั้น (ไม่มี /ห้อง เช่น "ป.2" ไม่ใช่ "ป.2/1") + เวลาที่แตะบัตร
  - ไม่แสดง: อวตาร, เลขบัตร
  - มีช่องค้นหาชื่อ

## 6. Modal ยกเลิกรายการบุฟเฟต์ (void/refund หลังจ่ายแล้ว)

ใช้โครงเดียวกับ modal ยกเลิกการจองใน Pre-Order:
- แสดงรายละเอียด transaction: ชื่อนักเรียน, ระดับชั้น, วันเวลาที่จ่าย, ยอดที่จ่าย
- dropdown เหตุผล: จ่ายผิดคน / จ่ายซ้ำโดยไม่ตั้งใจ / นักเรียนไม่ได้เข้าบุฟเฟต์จริง / อื่นๆ (ระบุเพิ่มเติม)
- ช่องกรอกรหัส supervisor (password, 4-6 หลัก) — รหัสผิด → error ใต้ช่อง ไม่ปิด modal
- ปุ่ม "ยืนยันยกเลิกรายการ" — disabled จนกว่าจะกรอกครบ
- เมื่อยกเลิกสำเร็จ → ตัดออกจากยอดสรุปภาพรวม (ข้อ 5) ทันที + log เหตุผล/ผู้อนุมัติ/เวลา
- ต้องตัดสินใจ (ถามทีม): เงินที่หักไปแล้วต้อง refund เข้าบัตรอัตโนมัติไหม หรือพนักงานจัดการนอกระบบ
