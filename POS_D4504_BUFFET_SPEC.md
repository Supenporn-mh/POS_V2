## 2. Flow

> **แก้ไขจากเวอร์ชันก่อนหน้า:** จากภาพหน้าจอจริงที่ได้รับมา ยอดเงินโชว์อยู่แล้วก่อนแตะบัตร (เช่น "ยอดบุฟเฟต์ ฿120.00") แปลว่าราคาถูกกำหนดไว้แล้วตั้งแต่ตอน**เลือกประเภทบุฟเฟต์** (เช่น cashier กดเลือก "บุฟเฟต์ ป.4-6" ซึ่งผูกราคาไว้ในตัวอยู่แล้ว) ไม่ใช่ระบบมารู้ยอดหลังแตะบัตรอย่างที่เข้าใจไว้ก่อนหน้า — แตะบัตร/สแกน QR คือขั้นตอน**ยืนยันตัวตน+ตัดเงิน**เท่านั้น ไม่ใช่ขั้นตอนคำนวณราคา

```
[เลือกประเภทอาหาร] → [เลือกประเภทบุฟเฟต์ที่ผูกราคาไว้แล้ว เช่น "บุฟเฟต์ ป.4-6" หรือ "บุฟเฟต์ คุณครู"]
  → [เลือกวิธีชำระเงิน: แตะบัตร / สแกน QR]

  ── แตะบัตร ──
  → หน้าแตะบัตร โชว์ยอดที่ผูกไว้แล้ว (เช่น ฿120.00) → แตะบัตรเพื่อยืนยันตัวตน+ตัดเงิน
  → [ชำระเงินสำเร็จ] แสดงชื่อ+ระดับชั้น+ยอดที่จ่าย

  ── สแกน QR (ใหม่) ──
  1. เลือกประเภทผู้ใช้งาน (คุณครู / นักเรียนกลุ่มไหน) — ถ้ายังไม่ได้เลือกจากขั้นตอนก่อนหน้า
  2. เลือกช่องทางชำระเงิน (PromptPay / e-wallet อื่นๆ)
  3. แสดง QR Code พร้อมยอดเงินที่ผูกไว้แล้ว + ระบุประเภทบุฟเฟต์ที่กิน
  4. ผลลัพธ์: ชำระสำเร็จ / ไม่สำเร็จ (หมดเวลา/ยกเลิก/เครือข่ายมีปัญหา)

[หน้าภาพรวมบุฟเฟต์] (staff)
  → filter วันที่
  → การ์ดสรุป: จำนวนคนทั้งหมด + ยอดรวม (รวมทั้งบัตรและ QR)
  → แยกตามกลุ่ม **อยู่ในตารางเดียวกัน** (ป.1-3, ป.4-6, ม.ต้น, ม.ปลาย, คุณครู) — ไม่แยก tab
  → กดแถว → [drill-down] รายชื่อในกลุ่มนั้น (ชื่อ + กลุ่ม + เวลา + วิธีชำระเงิน)
  → กดรายชื่อ (ถ้าต้องการยกเลิกรายการ) → [modal ยกเลิก]
```

## 2.1 หน้าจอเส้นทางสแกน QR (ใหม่)

| ขั้นตอน | เนื้อหา |
|---|---|
| เลือกวิธีชำระเงิน | ปุ่ม 2 ตัวเท่ากัน: "แตะบัตร" / "สแกน QR" — cashier เป็นคนเลือกให้นักเรียน/ครู |
| 1. เลือกประเภทผู้ใช้งาน | ปุ่มกลุ่ม: คุณครู, ป.1-3, ป.4-6, ม.ต้น/ปลาย (หรือแยกตามที่ระบบตั้งราคาไว้จริง) — ข้ามขั้นตอนนี้ได้ถ้าเลือกกลุ่มไว้แล้วจากตอนเลือกประเภทบุฟเฟต์ |
| 2. เลือกช่องทางชำระ | ใช้ list เดียวกับหน้าเติมเงิน (reference จริง): **PromptPay** (สแกน QR จ่ายได้เลย), **Credit/Debit Card**, **Alipay**, **WeChat Pay** |
| 3. แสดง QR Code | โชว์ยอดเงินตามกลุ่มที่เลือก + ชื่อประเภทอาหาร (บุฟเฟต์) + นับถอยหลัง **180 วินาที (3 นาที)** |
| 4. ผลลัพธ์ | สำเร็จ → เหมือนหน้าแตะบัตรสำเร็จ (เขียว+ติ๊กถูก) รวมยอดจ่าย/คงเหลือ/วันเวลา ตามข้อ 3.2 / **หมดเวลา → ยกเลิกรายการทันที** (ไม่ auto-retry) แคชเชียร์ต้องกดเริ่มรายการใหม่เองจากหน้าเลือกวิธีชำระเงิน |

หมายเหตุ: ตัดสินใจแล้วว่าเลือกประเภทอาหารก่อนเสมอ ไม่ว่าจะจ่ายด้วยบัตรหรือ QR — ลำดับคือ **ประเภทอาหาร → วิธีชำระเงิน** ไม่ใช่กลับกัน

## 3. หน้าจอ 1 (student-facing) — บุฟเฟต์

### 3.1 Layout มาตรฐาน (อ้างอิงจาก reference จริง — ใช้ layout นี้เป๊ะ)

โครงสร้าง: แบ่งซ้าย-ขวาด้วยเส้นคั่นแนวตั้ง

**หัวการ์ด:** ข้อความบอกหน้า (เช่น "หน้าแตะบัตร — เมนูบุฟเฟต์") มุมซ้ายบน + ไอคอนเมนู "..." มุมขวาบน

**ฝั่งซ้าย:**
- Badge ประเภทอาหาร ("บุฟเฟต์") เป็น pill สีพื้นอ่อน มุมบนสุด
- รายการข้อมูล: เลขบัตร (–), ชื่อนักเรียน (–), ยอดก่อนชำระ (0.00) — ทุกอย่างว่างจนกว่าจะแตะบัตร
- ล่างสุด: ไอคอนเงินสด + 0.00, ไอคอน QR/wallet + 0.00, วันที่ (dd-mm-พ.ศ.)

**ฝั่งขวา:**
- ปุ่ม "ประวัติ" มุมขวาบน (icon นาฬิกาย้อนกลับ)
- กึ่งกลาง: "แตะบัตรเพื่อชำระเงิน" (ตัวหนา ใหญ่) + "ยอดบุฟเฟต์ ฿{จำนวน}" ใต้ลงมา
- ไอคอนบัตรสี่เหลี่ยมมุมโค้งขนาดใหญ่ พื้นหลังสีเข้ม (ใช้ `--text-primary` หรือสี primary เข้ม) ตรงกลาง

> ใช้ layout นี้เป็น template เดียวกันกับทุกประเภทอาหาร ไม่ใช่ทำใหม่แยกแต่ละเมนู — เปลี่ยนแค่ badge ชื่อประเภทกับยอดเงิน

### 3.2 หลังแตะบัตรสำเร็จ
- ไอคอนติ๊กถูกสีเขียว
- "ชำระเงินสำเร็จ"
- ชื่อนักเรียน/ครู + ระดับชั้น/กลุ่ม + ประเภท "บุฟเฟต์"
- **ยอดที่จ่าย** (ตัวเลขใหญ่ สีธีมหลัก)
- **ยอดเงินคงเหลือในระบบหลังหักแล้ว** (เช่น "คงเหลือ ฿95.00" — ถ้าติดลบให้โชว์เป็นค่าลบตรงๆ ไม่ปัดเป็น 0)
- **วันที่และเวลาที่ทำรายการ** (เช่น "13 ก.ค. 2569 · 12:04 น.")

### 3.3 จุดเข้าถึง "ภาพรวมบุฟเฟต์" (แก้ไขตำแหน่ง)

> เดิมปุ่ม "ประวัติ" อยู่มุมขวาบนของ**หน้าแตะบัตร** — ใช้ไม่ได้แล้วเพราะตอนนี้มี 2 เส้นทางการจ่าย (แตะบัตร/สแกน QR) ไม่ใช่ทุกรายการจะผ่านหน้าแตะบัตร
>
> **ย้ายไปไว้ที่หน้าเลือกประเภทบุฟเฟต์แทน** (หน้าแรกสุดของ flow ก่อนแยกไปแตะบัตรหรือสแกน QR) เพราะเป็นหน้าเดียวที่ทุก transaction ต้องผ่านแน่นอน ไม่ว่าจะจ่ายด้วยวิธีไหน — ปุ่ม "ภาพรวมบุฟเฟต์" อยู่มุมขวาบนของหน้านี้แทน

### 3.4 Edge cases ที่ใช้ร่วมกับ Pre-Order ได้ (ใช้ mapping เดียวกัน)
- บัตรไม่รู้จัก / บัตรถูกระงับ / ระบบออฟไลน์ / อ่านบัตรไม่ผ่าน (ฮาร์ดแวร์) — ใช้ข้อความและสีเดียวกับที่กำหนดใน `POS_D4504_TAP_TO_COLLECT_SPEC.md` ข้อ 4 และ 4.5 ได้เลย ไม่ต้องออกแบบใหม่

### 3.5 Edge case เฉพาะบุฟเฟต์ (ยืนยันกับทีมแล้ว)

- **แตะบัตรซ้ำในมื้อเดียวกัน** → แสดง error ทันที "มื้อนี้ชำระบุฟเฟต์ไปแล้ว" พร้อมเวลาที่จ่ายครั้งแรก **ไม่มีปุ่มยืนยันจ่ายซ้ำ** (ตัดออกจากเวอร์ชันก่อนหน้า) เพราะกฎคือ: ถ้ามีคนอื่นต้องการจ่ายด้วยบัตรใบเดียวกัน (เช่น กรณีลืมบัตรตัวเอง) **ให้ใช้วิธีสแกน QR แทน ไม่ใช่แตะบัตรซ้ำ**

- **ยอดเงินไม่พอจ่าย (นิยามใหม่ — ระบบอนุญาตติดลบได้ถึงเพดานที่ตั้งไว้)**
  ระบบให้บัตรติดลบได้ถึงเพดานหนึ่ง (เช่น ติดลบได้ไม่เกิน ฿200) การเช็ก "พอจ่ายไหม" ไม่ใช่เช็กว่า `ยอดคงเหลือ >= ยอดบุฟเฟต์` แต่เป็นเช็กว่า `ยอดคงเหลือ - ยอดบุฟเฟต์ >= -เพดานติดลบ`
  - ถ้าจ่ายแล้วยังไม่เกินเพดานติดลบ → **ผ่านได้ปกติ** (ยอดคงเหลือหลังหักจะติดลบได้ ไม่ใช่ error)
  - ถ้าจ่ายแล้วจะเกินเพดานติดลบที่ตั้งไว้ → **บล็อกการแตะบัตร** แสดง "ยอดเงินไม่เพียงพอ (เกินวงเงินติดลบที่กำหนด)" + **ต้องเปลี่ยนไปใช้วิธีสแกน QR แทน** (ไม่มีทางแตะบัตรผ่านได้ในเคสนี้)
  - เพดานติดลบ (ตัวอย่าง ฿200) เป็นค่าที่ตั้งได้ในระบบ ต้องดึงจาก config ไม่ hardcode

## 4. จอ 2 (customer display) — บุฟเฟต์

> **แก้ไข:** จอ 2 ต้อง **mirror เนื้อหาเดียวกับจอ 1 (cashier) ทุกเคส** ไม่ใช่ fallback กลางแบบที่เคยออกแบบไว้ก่อนหน้า — ลูกค้าควรเห็นสถานะ/ข้อความเดียวกับที่ cashier เห็น รวมถึง edge case ทุกเคส (บัตรถูกระงับ, เงินไม่พอ, ระบบออฟไลน์ ฯลฯ) เพียงแต่ตัดข้อมูลอ่อนไหวระดับ log/error code ทางเทคนิคออก (เช่น error code เครื่องขัดข้อง) เหลือแค่ข้อความที่มนุษย์อ่านเข้าใจ

- ก่อนแตะ: "แตะบัตรเพื่อชำระเงิน" + ยอดบุฟเฟต์ที่ผูกไว้แล้ว (เหมือนจอ 1 เป๊ะ)
- หลังแตะสำเร็จ: "ชำระเงินสำเร็จ" + ยอดที่จ่าย
- ทุก edge case: ใช้ข้อความเดียวกับจอ 1 (ดูตารางเต็มใน `POS_D4504_TAP_TO_COLLECT_SPEC.md` §4 สำหรับ pattern ข้อความ แล้ว adapt เป็นบริบทบุฟเฟต์)

## 5. หน้าภาพรวมบุฟเฟต์ (staff)

- ช่องเลือกช่วงวันที่
- การ์ดสรุป: จำนวนคนเข้าบุฟเฟต์ทั้งหมด + ยอดรวม (รวมทุกวิธีชำระเงิน)
- แยกตามกลุ่ม **อยู่ในตารางเดียวกันทั้งหมด** (ตัดสินใจแล้ว: ไม่แยก tab ครู/นักเรียน):
  ```
  ป.1-3     → 62 คน
  ป.4-6     → 71 คน
  ม.ต้น     → 35 คน
  ม.ปลาย    → 16 คน
  คุณครู    → 8 คน
  ```
  แต่ละแถวโชว์ ชื่อกลุ่ม, ราคา/คน, จำนวนคน, ยอดรวมของกลุ่มนั้น
- กดแถว → drill-down แสดงรายชื่อในกลุ่มนั้น:
  - แสดง: ชื่อ + กลุ่ม (ระดับชั้น ไม่มี /ห้อง เช่น "ป.2" ไม่ใช่ "ป.2/1") + เวลาที่ชำระ + **วิธีชำระเงิน** (แตะบัตร / สแกน QR)
  - ไม่แสดง: อวตาร, เลขบัตร
  - มีช่องค้นหาชื่อ
  - ปุ่ม/action "ยกเลิกรายการ" ต่อแถว **แสดงเฉพาะรายการที่วิธีชำระเงินเป็น "แตะบัตร"** เท่านั้น — รายการที่จ่ายด้วย QR ไม่มีปุ่มนี้ (ดูเหตุผลในข้อ 6)

### 5.1 ข้อมูลเพิ่มเติมที่ต้องมีในประวัติ/ภาพรวม (จากการเพิ่มช่องทาง QR)

| ข้อมูล | เหตุผล |
|---|---|
| วิธีชำระเงิน (บัตร/QR) | แยกสรุปได้ว่าจ่ายด้วยบัตรกี่คน QR กี่คน |
| ช่องทางที่ใช้ (PromptPay/Credit-Debit Card/Alipay/WeChat Pay) | ใช้ reconcile กับผู้ให้บริการแต่ละเจ้า |
| สถานะรายการ (สำเร็จ/ไม่สำเร็จ-หมดเวลา) | รายการไม่สำเร็จไม่นับเข้ายอดสรุป แต่เก็บ log ไว้ debug ได้ |
| เลขอ้างอิงธุรกรรม (transaction ref จาก QR provider) | ใช้ตรวจสอบย้อนหลังกรณีเงินไม่เข้า |

## 6. Modal ยกเลิกรายการบุฟเฟต์ (void/refund หลังจ่ายแล้ว)

> **สำคัญ: ยกเลิกได้เฉพาะรายการที่จ่ายด้วยการแตะบัตรเท่านั้น**
> - **แตะบัตร → ยกเลิกได้** เพราะเงินอยู่ในระบบของโรงเรียนเอง คืนเข้าบัตรได้โดยตรง
> - **สแกน QR → ยกเลิกไม่ได้** เพราะเป็นการจ่ายผ่านธนาคาร/ผู้ให้บริการ e-wallet โดยตรง ระบบไม่มีกลไก refund เงินคืน
>
> ผลต่อ UI: ปุ่ม "ยกเลิกรายการ" ในหน้า drill-down (ข้อ 5) และ modal นี้ **ต้องไม่แสดง** (หรือแสดงแบบ disabled พร้อมคำอธิบาย) เมื่อรายการนั้นมีวิธีชำระเงินเป็น "สแกน QR" — เช็กจาก field วิธีชำระเงินก่อนแสดงปุ่มเสมอ ถ้าพนักงานพยายามกดจากรายการ QR ให้ขึ้นข้อความ "รายการนี้ชำระผ่าน QR ไม่สามารถยกเลิก/คืนเงินผ่านระบบได้ กรุณาติดต่อธนาคาร/ผู้ให้บริการโดยตรง"

Layout ตาม reference จริง (โครงเดียวกับ modal ยกเลิกฝั่ง Pre-Order แต่ยึดข้อมูลบุฟเฟต์) — **ใช้กับรายการที่จ่ายด้วยแตะบัตรเท่านั้น:**

- **หัว:** ไอคอนใบเสร็จในกล่องสีพื้น + "รายละเอียดการชำระเงิน" + เลขอ้างอิงธุรกรรม (เช่น `BUFF00214`) + badge สถานะ "ชำระแล้ว" มุมขวา + ปุ่มปิด (X)
- **ข้อมูลธุรกรรม:** ชื่อนักเรียน/ครู + ระดับชั้น/กลุ่ม, วันที่-เวลา-วิธีชำระ (แตะบัตร/QR), รายการ "บุฟเฟต์ {กลุ่ม} · ฿{ยอด}"
- **เหตุผลการยกเลิก \*** — ปุ่มเลือกแบบ pill (เลือกได้ทีละอัน): "จ่ายผิดคน" / "จ่ายซ้ำโดยไม่ตั้งใจ" / "ไม่ได้เข้าบุฟเฟต์จริง" / "อื่นๆ"
- **รายละเอียดเพิ่มเติม** — textarea อิสระ (ไม่บังคับ ยกเว้นเลือก "อื่นๆ" ต้องกรอก)
- **รหัสยืนยัน (PIN) \*** — ช่องกรอกแบบ dot input 4 หลัก พร้อมคำอธิบาย "กรอก PIN ของคุณเพื่อยืนยันการยกเลิก"
- **ข้อความเตือน** สีแดงอ่อน: "การยกเลิกนี้ไม่สามารถย้อนกลับได้ และจะถูกบันทึกไว้ในระบบ"
- **ปุ่มท้าย:** "ย้อนกลับ" (secondary) และ "ยืนยันยกเลิก" (สีแดง, primary danger) วางคู่กัน

Logic:
- ปุ่ม "ยืนยันยกเลิก" disabled จนกว่าจะเลือกเหตุผล + กรอก PIN ครบ 4 หลัก
- PIN ผิด → error ใต้ช่อง ไม่ปิด modal
- ยืนยันสำเร็จ → คืนเงินเข้าบัตรนักเรียน/ครูโดยอัตโนมัติ + ตัดออกจากยอดสรุปภาพรวม (ข้อ 5) ทันที + log เหตุผล/รายละเอียดเพิ่มเติม/ผู้อนุมัติ/เวลา

## 7. สถานะการยืนยันกับทีม

**ยืนยันแล้ว (ใช้ตามนี้ได้เลย):**
- [x] คืนเงินกรณีย ยกเลิกรายการที่จ่ายด้วยบัตร → **คืนทันทีแบบ real-time**
- [x] แตะบัตรซ้ำในมื้อเดียวกัน → ขึ้น error เสมอ ไม่มีทางจ่ายซ้ำผ่านการแตะบัตร ถ้าจะจ่ายให้คนอื่นด้วยบัตรเดียวกันต้องสลับไปสแกน QR
- [x] ยอดเงินไม่พอจ่าย → หมายถึงเกินเพดานติดลบที่ระบบตั้งไว้ (เช่น ติดลบได้ไม่เกิน ฿200) ไม่ใช่ต้องมียอดพอเป๊ะ — เกินเพดานเมื่อไหร่ต้องสลับไปสแกน QR
- [x] ช่องทางชำระเงิน (QR/สแกนจ่าย) → PromptPay, Credit/Debit Card, Alipay, WeChat Pay (ตาม reference จริงจากหน้าเติมเงิน)
- [x] เวลานับถอยหลัง QR → 180 วินาที (3 นาที) หมดเวลาแล้ว **ยกเลิกรายการทันที** ไม่ auto-retry แคชเชียร์กดเริ่มใหม่เอง
- [x] **ข้อมูลหน้าภาพรวม/สรุปบุฟเฟต์ทั้งหมดในเอกสารนี้เป็น mock ได้ก่อน** เพราะของจริงดึงมาจาก backoffice — ไม่ต้องรอ API จริงถึงจะเริ่ม build UI

**ยังต้องยืนยันเพิ่ม:**
